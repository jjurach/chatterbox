**Title:** Epic structure
**Project:** chatterbox

The project involves 12 different epics focused on creating an Alexa-like device using an Espressif Box 3B. We want to apply large-scale orchestration experimentation with targets, goals, and milestones. Each epic will have a spec listing expectations and task details, and the individual tasks and their dependencies will be represented using the @beads/bd tool. Beads will determine the next work which agents can pick up. It is feasible to run unrelated epics in parallel -- e.g. to work device and backend projects by different agents concurrently.

Epic 1 will establish OTA cadence, introduce a tool which can push selected images out to selected devices, and confirm agent can autonomously deploy new builds to the device. Agent will guide human to configure the OTA update of the chatterbox firmware.  Implement a state machine within the Box3B (or harden triggers within the ESPHome .yaml to "loop" correctly and always to be in a verifiable state) to cycle through six different screens (orange, purple, blue, red, yellow, green, back to blue, etc.) in a repeating sequence with delays between transitions. Success for Epic 1 requires implementing this state machine, which may involve ESPHome YAML or embedded logic. It also includes creating graphics with "puppy images" in the upper left and clear letters representing the state in the lower right. Additionally, it involves setting up visual OCR recognition on the state letters so agents can interact with the device, play generated text, and process outputs to improve system quality. We will document these 6 screens to have names, and we will provide sets of graphics (i.e. themed) which will be incorporated on each of these screens.

Epic 2 focuses on observability, using both video monitoring to verify screen content and rendering speed, plus serial connections for the ESPHome application to send information to the agent, allowing it to track the state machine independently of visual confirmation. Agent will guide human through setting up access to home assistant logs. This is setting up testing and observation patterns which will be repeated across all subsequent epics. 

Epic 3 handles chatterbox device network connectivity, ensuring the Chatterbox device has Wi-Fi and broadcasts its presence. The device needs to listen on the correct port for Wyoming protocol connections from Home Assistant. This epic specifically addresses the transition from purple (Wi-Fi connected but no Home Assistant connection) to blue (Home Assistant connected). By the end of this epic, the chatterbox firmware should provide detailed logging of packets received by home assistant through the serial connection. Let's explore API actions agents can initiate on home assistant which will induce packet activity on the chatterbox.

Epic 4 validates Wyoming connections, explores simulators on both sides of the protocol, handles PCM transfers, and investigates streaming to Whisper versus buffering into a larger waveform. This epic will handle both input and output PCM streams (i.e. whisper and piper integration), and will lean into a variety of simulation tools to iterate through each of the different roles of client and server within this system. Here we will configure Home Assistant, to register all features the chatterbox backend will support: stt, tts, assist, etc.

Epic 5 focuses on LLM integration and tooling, possibly including web search or weather search capabilities. Whereas, epic 4 will likely encode the hard-coded text message "your message was received", this epic is all about involving an LLM loop and lang-graph. Implement tooling for: weather and news.

Epic 6 addresses buttons and user interaction, as ESPHome currently doesn't support the Box 3B's touch features. This involves implementing touch sensitivity and mapping screen presses to button actions in the state machine. Different screens will have different buttons - the blue screen will have a wake-up button, the red screen will have pause and stop buttons, the yellow screen may have a "never mind" button, and the green screen will have a stop button.

Epic 7 focuses on recording, where the Box 3B transmits PCM packets through the Wyoming protocol and Home Assistant to the Chatterbox backend, which accumulates and buffers them for submission to Whisper. This epic also involves ensuring the backend correctly processes these packets.

Epic 8 handles receiving home assistant responses, where the device transitions from the yellow waiting state to the green rendering state when receiving PCM packets from the backend. This involves implementing the audio playback functionality to render the PCM packets into sound. We will also introduce subtle beep or click sound on transition from blue to red (to alert recording) and from green to blue (to signify answer complete). Let's produce a few different sounds, and allow the sound choices to be specified in the ESPHome .yaml

Epic 9 is a testing epic whereby agent will initiate wake word with device by playing audio on the local server. We will extend the video capture and serial logging techniques to include audio features. Agent will validate wake word settings and optimize device responsiveness. Agent will validate adherence to the orange, purple, blue, red, yellow, green, blue, ... sequence.

Epic 10 implements a 5 second "red step" after green -- any speech detected within that 5 seconds will trigger that same 5 second duration prepended to the following recording be transmitted to the server as an interactive exchange. In other words, if the user starts talking to the agent within 5 seconds after the agent speaks, then the agent will listen and continue the interaction. If there is no speech detected within that 5 seconds, then the device switches to blue state, requiring wake word or button press.

Epic 11 will iterate on reliability and cost optimization techniques. A list of 5-10 different LLM provider and local LLM options are available to cycle through on failure. More attention to context length and tracking which llm profile (or whatever the appropriate abstraction is) allows wihch context lengths. When all LLMs fail, then let python guess with some basic heuristic analysis or maybe just render a default response.

Epic 12 will be about archiving/compacting any history, reviewing documentation, reducing clutter and repetition within the project, setting the project up for long-term maintenance.
